# Migrating to cross-service enqueue

The cross-service-enqueue work in soniq replaces two old shapes with
new ones. This is a breaking change. soniq is pre-1.0 so there is no
backwards-compatibility shim; existing call sites must be rewritten.

This guide walks you through the rewrite by hand and describes the
`soniq migrate-enqueue` codemod that automates the bulk of it.

## What changed

This release adds **cross-service enqueue**: a service that owns the
producer side can enqueue jobs that a different service (different
repo, different container) consumes, both pointing at the same
Postgres database. The new surface is layered on top of the existing
single-repo API; **most code does not need to change**.

| Area | Status |
| --- | --- |
| `@app.job` decorator | Unchanged. Task names default to `f"{module}.{qualname}"` (Celery-style); pass `name=` explicitly for cross-service deployments. |
| `await app.enqueue(my_func, x=1)` | Still supported. Three input shapes: callable, string name, or `TaskRef`. |
| `SONIQ_ENQUEUE_VALIDATION` | New setting (default `"strict"`). Governs how `enqueue("string-name", ...)` handles a name not registered locally. |
| `task_ref()` / `TaskRef` | New typed-constant for shared stub packages between producer and consumer. |
| `soniq migrate-enqueue` | New CLI codemod for projects that want to switch to the by-name shape in bulk. |

## Manual rewrite

**For most single-repo code, nothing needs to change.** The
decorator works as before, the callable form of `enqueue` is back,
and task names are derived `module.qualname`-style:

```python
# unchanged from earlier versions
@app.job()
async def send_welcome(user_id: int):
    ...

await app.enqueue(send_welcome, user_id=42)
```

For **cross-service** deployments (producer and consumer in
different repos), use the by-name form:

```python
await app.enqueue("users.send_welcome", args={"user_id": 42})
```

Pair that with an explicit `name=` on the decorator so the wire
identifier is stable across refactors:

```python
@app.job(name="users.send_welcome")
async def send_welcome(user_id: int):
    ...
```

For **typed cross-repo stubs**, use a `TaskRef`:

```python
from soniq import task_ref

send_welcome_ref = task_ref(
    name="users.send_welcome",
    args_model=WelcomeArgs,
)

await app.enqueue(send_welcome_ref, args={"user_id": 42})
```

All three shapes route through the same `enqueue` method;
disambiguation is by the type of the first argument (callable, str,
or `TaskRef`).

Enqueue-time options (`queue`, `priority`, `scheduled_at`, `unique`,
`dedup_key`, `connection`) stay at the top level; only function args
collapse into the `args=` dict:

```python
# before
await app.enqueue(send_welcome, user_id=42, queue="urgent", priority=5)

# after
await app.enqueue(
    "users.send_welcome",
    args={"user_id": 42},
    queue="urgent",
    priority=5,
)
```

## Using the codemod

The `soniq migrate-enqueue` codemod is **only useful** if you want
to switch a project from the callable form to the by-name form (for
example, because you're carving the consumer out into a separate
service and want stable wire-protocol names for the producer side).
**Single-repo projects don't need to run it** — the callable form
keeps working.

```bash
pip install -U soniq
soniq migrate-enqueue --help
```

Supply a mapping in `migrate-enqueue.toml` listing the canonical task
names:

```toml
[names]
send_welcome = "users.send_welcome"
charge_subscription = "billing.charge_subscription.v1"
generate_report = "reports.generate.v1"
```

Then run:

```bash
soniq migrate-enqueue path/to/your/project
```

The codemod rewrites each `app.enqueue(callable, **kwargs)` to
`app.enqueue("name", args={...})` using the names from the config.
Without a mapping, the codemod refuses to invent canonical names:
call sites with no mapping print a per-site diagnostic and are left
untouched (exit code 2).

Pass `--use-derived-names` to fall back to `f"{module}.{qualname}"`
when no explicit mapping is configured. This matches what `@app.job`
already does internally and is fine for single-repo conversions.

## Naming convention

The default `SONIQ_TASK_NAME_PATTERN` accepts dotted lowercase
identifiers like `billing.invoices.send.v2`. We recommend the dotted-
suffix versioning pattern (`name.v2`, not `name_v2`) for cleaner
prefix-based metric grouping and routing.

## Producer / consumer split

Producer-vs-consumer is a deployment convention, not a class hierarchy
or constructor flag. The same `Soniq` runs in both roles; the deployment
decides whether to import handler modules and whether to call
`run_worker(...)`. See [Deployment shapes](../production/deployment-shapes.md)
for the patterns.

## Validation modes

`SONIQ_ENQUEUE_VALIDATION` (default `strict`) controls what
`enqueue()` does when the name is not registered locally.

- **strict** (default): raise `SONIQ_UNKNOWN_TASK_NAME`, no row
  written. Single-repo / monolith deployments.
- **warn**: rate-limited WARN per unknown name; enqueue proceeds.
  Producer services that need lenience.
- **none**: silent passthrough. Pure producers with no local
  registry by design.

Producer services that enqueue names they don't register should set
`SONIQ_ENQUEUE_VALIDATION=warn` (or `=none`) explicitly in their
deployment environment.

## What did not change

- The wire format (`job_name TEXT`, `args JSONB/JSON`).
- Worker dispatch, retry policy, dead-letter behaviour.
- The `dedup_key` / `unique` deduplication helpers.
- The `connection=` transactional-enqueue path.
- The `@app.periodic(...)` decorator (it derives `name=func.__name__`
  if you don't pass one explicitly, since periodic jobs are by
  convention single-repo bookkeeping).

## Delivery semantics

soniq is **at-least-once**, not exactly-once. Handlers must be
**idempotent** because retries, transactional rollback edge cases,
and worker crashes can cause re-delivery. See the
[cross-service guide's idempotency section](../guides/cross-service-jobs.md#at-least-once-delivery-and-idempotency)
for the recommended dedup-key pattern.
